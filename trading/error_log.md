### ERROR-001: buy()参数位置错误（P0-致命）

**发现时间**：2026-05-26 11:54
**影响范围**：`lobster_intraday_patrol.py` 所有买入操作

**错误代码**：
```python
# 修复前
buy(code, name, q['price'], reason, dim, pos_pct, sector=sector)

# buy()函数签名
def buy(code, name, price, reason, dimension, up_count=None, position_pct=None, ...):
```

**根本原因**：
- `pos_pct` (10/15) 被当作 `up_count` 传入
- `position_pct` 参数默认为 `None`
- 导致仓位计算全部错误

**修复方案**：
```python
# 修复后
buy(code, name, q['price'], reason, dim, position_pct=pos_pct, sector=sector)
```

**验证方法**：
- 检查 `simulated_trading.py` 的 `buy()` 调用处
- 验证 `position_pct` 正确传入

**状态**：✅ 已修复（11:54）

**教训**：
1. Python混合位置参数和关键字参数时，位置参数会误传
2. 必须用关键字参数明确意图，避免位置依赖
3. 函数签名变更时，所有调用处必须同步更新

**教训**：
1. **第三方API字段必须有映射表**，不能硬编码索引

### ERROR-003: is_late判断逻辑错误（P2-中等）

**发现时间**：2026-05-26 11:54
**影响范围**：`lobster_intraday_patrol.py` 尾盘判断

**错误代码**：
```python
# 修复前
now_min = datetime.now().hour * 60 + datetime.now().minute
if now_min >= 45:  # 任意小时的45分都触发
    is_late = True

# 修复后
if now_min >= 14*60+45:  # 仅14:45后触发
    is_late = True
```

**根本原因**：
- 开发者理解错误：`45` 应该是"14:45"，但写成了"任意:45"
- 因cron只在xx:00/xx:30/14:45触发，实际运行时未触发BUG

**修复方案**：
```python
# 更清晰的写法
now = datetime.now()
is_late = (now.hour >= 14 and now.minute >= 45)
```

**验证方法**：
- 单元测试：测试 14:44→False, 14:45→True, 15:00→True

**状态**：✅ 已修复（11:54）

**教训**：
1. 时间判断用`hour*60+minute`容易出错，建议用`datetime`对象比较
2. 魔法数字（magic number）必须有注释说明

### ERROR-004: 仓位单位BUG（5成 vs 5%）（P1-严重）

**发现时间**：2026-05-26 12:44
**影响范围**：`lobster_intraday_patrol.py` `adjust_position_pct()` 函数

**错误代码**：
```python
# 修复前
def adjust_position_pct(dim, base_pct, pos_limit, positions):
    # pos_limit = 5 (从config读取，表示5成)
    # 但函数里把 5 当成 "5%" 计算
    max_pct = pos_limit  # 5 → 误以为5%
    # ...

# 调用处
pos_limit = config['emotion_matrix']['冰点']['max_positions']  # = 5
adj_pct = adjust_position_pct(dim, base_pct, pos_limit, positions)
```

**根本原因**：
- Config中 `max_positions: 5` 表示"5成"
- 但函数里当成 "5%" 计算
- 导致冰点单仓只有5%而不是10%

**错误影响**：
- 冰点期单仓应该是10%（5成÷5只）
- 实际只有5%（5%÷5只）→ **仓位少了一半**

**修复方案**：
```python
# 修复后
def adjust_position_pct(dim, base_pct, pos_limit_cheng, positions):
    """
    pos_limit_cheng: 成数，如5表示5成=50%
    """
    pos_limit_pct = pos_limit_cheng * 10  # 5成 → 50%
    max_pct = pos_limit_pct / len(positions)  # 50% ÷ 5只 = 10%
    # ...
```

**验证方法**：
- 单元测试：
  - 冰点(5成)：已持0只→10%, 已持5只→0% ✅
  - 活跃期(7成)：已持0只→10%, 已持7只→0% ✅

**状态**：✅ 已修复（12:44）

**教训**：
1. **参数单位必须在函数签名/文档中明确**
2. Config的语义（5=5成）必须和代码语义（5=5%）对齐
3. 建议用枚举或常量：`CHENG_5 = 50%`, `CHENG_7 = 70%`

步骤8: 立即推送用户  ← 优先

**验证方法**：

### ERROR-007: 盘中巡检Cron表达式错误（P1-严重）

**发现时间**：2026-05-26 13:07（用户反馈："下午从1点开始的盘中巡查没有了"）
**影响范围**：盘中巡检cron任务（13:00-14:30无执行）

**错误配置**：
```
schedule: cron */30 10-11 * * 1-5
```

**错误分析**：
- `*/30` = 每30分钟（0分和30分）
- `10-11` = 仅10点和11点
- **不包含**下午13点、13:30、14:00、14:30

**正确配置**：
```
schedule: cron 0,30 10,13-14 * * 1-5
```

**错误影响**：
- 13:00-14:30 无任何盘中巡检
- 用户收不到下午的监控报告
- 可能错过买卖点

**修复操作**：
```bash
openclaw cron edit 93bdca91 --cron "0,30 10,13-14 * * 1-5"
```

**验证方法**：
- `openclaw cron show 93bdca91` 确认表达式
- 等待13:30自动执行，确认推送到达

**状态**：✅ 已修复（13:15）/ ⚠️ 验证待完成（等13:30）

**教训**：
1. **Cron表达式必须验证**：创建/修改后必须检查下次运行时间
2. **小时范围要完整**：`10-11`只包含2个小时，不包含13-14
3. **逗号分隔多个时段**：`10,13-14` = 10点+13点+14点

## 系统性问题总结

### 问题1：缺乏字段映射表（ERROR-002）

**现象**：
- 腾讯API字段索引硬编码在代码中
- 不同脚本可能有不同的索引理解
- 没有统一的字段映射常量

**建议**：
```python
# scripts/tencent_api.py
TENCENT_API_FIELDS = {
    'NAME': 1,
    'CODE': 2,
    'PRICE': 3,
    'CLOSE': 4,
    'OPEN': 5,
    'HIGH': 6,
    'LOW': 7,
    'VOLUME': 8,
    'AMOUNT': 9,
    # ...
}

def parse_stock_data(raw_string):
    p = raw_string.split('~')
    return {
        'price': float(p[TENCENT_API_FIELDS['PRICE']]),
        'close': float(p[TENCENT_API_FIELDS['CLOSE']]),
        # ...
    }
```
